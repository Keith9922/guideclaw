# 引路虾冲奖优化路线图

这份路线图只关注一件事：

**如果继续打磨，哪些事情最值得做，最有可能把当前版本从“强原型”推到“高奖项竞争态”。**

## 当前判断

当前版本已经具备：

- 完整的前后端闭环
- 多 Agent 工作流
- OpenClaw 真实执行
- 项目级知识库
- PDF 上传与结果汇总
- 可直接演示的最终交付页

所以当前定位应当是：

**可参赛、可演示、可入围的高完成度版本。**

要继续冲更高奖项，应该优先做“增强可信度”和“增强交付感”的事，而不是继续发散加很多功能。

## 一、第一优先级：强化科研级证据链

这是最值钱的一条线。

当前已有：

- 文献来源
- DOI / URL
- 项目知识库
- PDF 入库
- 轻量 RAG

还可以继续补：

- PDF 正文级 chunk 检索
- 更稳定的页面 / 段落定位
- GROBID 结构化解析
- 更强的 rerank
- PaperQA 级正文证据回溯

为什么最重要：

- 它决定评委是否相信系统不是“会说”，而是真的“有依据”
- 它决定推荐阅读、缺口判断、方案建议的可信度
- 它是你和普通“论文问答工具”拉开差距的核心

## 二、第二优先级：强化结果汇总的成品感

结果汇总已经出现了，但还可以继续往“导师汇报材料”方向走。

建议优化：

- 更强的章节层次
- 更明确的“最推荐切入点”
- 更像一周执行计划的行动清单
- 更强的 PDF 版视觉层次
- 更明确的“当前证据充分 / 不足”的标记

目标不是加功能，而是让评委一眼觉得：

**这个系统真的能把科研新人带进门。**

## 三、第三优先级：把安全与边界做成显性能力

比赛明确强调：

- 本地运行
- 不暴露公网
- 数据安全
- 插件来源可信
- 行为边界透明

当前技术上基本符合，但产品表达还不够显性。

建议补一页或一块明确说明：

- 当前运行模式
- 数据存储位置
- 用户上传 PDF 的边界
- OpenClaw 的本地执行方式
- 系统不会代替用户做最终学术判断

为什么重要：

- 这会让评委对“可控性”更放心
- 能明显提升成熟度感

## 四、第四优先级：把追问链路继续做强

现在已经有“继续追问”，这是对的。

后面可以继续增强：

- 追问自动挂接到结果汇总
- 追问结果自动影响下一轮推荐切入点
- 追问结果与已有文档形成版本链
- PI 明确标注“这轮为什么继续派这个 Agent”

这样多轮推进感会更强。

## 五、第五优先级：移动端 / 飞书作为加分项

飞书不是当前主线，但它是很好的“眼前一亮”项。

适合做的不是完整工作台，而是轻量能力：

- 查看项目当前状态
- 查看最新结果汇总
- 接收研究摘要通知
- 发起继续追问
- 查看下一步行动

这样评委会更容易理解：

**Web 是深度工作台，飞书是轻量远程操控入口。**

## 六、不建议继续花大力气的方向

下面这些方向现在不是最优先：

- 再多堆一批角色
- 再做很多不必要的 skills 展示
- 再拆更多页面
- 做过度复杂的前端花活
- 为了炫技去做很重的多 Agent 自主循环

原因很简单：

比赛看的是“实际价值”和“可信度”，不是系统复杂度。

## 七、建议按这个节奏推进

### 阶段 1：可信度增强

- 强化 PDF 正文证据回溯
- 优化知识检索质量
- 提升推荐阅读相关性

### 阶段 2：交付物增强

- 继续打磨结果汇总页
- 提升 PDF 版样式
- 强化行动清单与推荐切入点

### 阶段 3：产品成熟度增强

- 安全边界展示
- 更清晰的运行说明
- 更清晰的项目状态与版本链

### 阶段 4：加分项

- 飞书轻量入口
- 更高级的引用可视化
- 可选的导师汇报版导出

## 八、最应该记住的原则

后续优化不要偏航。

真正该坚持的方向是：

- 不是让它看起来“更像 AI”
- 而是让它看起来“更像一个真的能帮科研新人推进研究工作的系统”

如果只能继续做少量工作，优先级永远是：

1. 证据链
2. 结果汇总
3. 安全边界
4. 追问链路

这条路线最有希望继续把当前版本推到更高奖项竞争态。
