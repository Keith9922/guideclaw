## 这版最值得讲的点

### 1. 产品定位

引路虾不是“论文问答工具”，而是一个**科研工作流系统**：

- `PI` 先定义研究焦点和关键问题
- `文献助理` 负责收集候选论文与引用源
- `选题分析员` 输出优先研究缺口
- `方案设计师` 收成一版可讨论方案
- `组会秘书` 沉淀纪要、待办与下一步

### 2. OpenClaw 在哪里起作用

当前真实链路是：

```text
Web 工作台 -> FastAPI -> OpenClaw CLI（按需执行）-> workspace skills -> Markdown 结果回写
```

职责边界：

- `引路虾 Web + API`
  - 项目管理
  - 知识库与 PDF 解析
  - RAG 检索
  - 工作流编排
  - 成果卡和 Markdown 持久化
- `OpenClaw`
  - 角色执行入口
  - workspace skills 运行
  - 本地 agent turn

### 3. 评委最容易感知到的价值

- 从一个模糊话题出发，快速拉起研究主线
- 每一轮输出都会沉淀，不是一次性聊天
- 成果卡带引用数量、链接、DOI 和证据片段
- PDF 可以进入项目知识库，后续检索会围绕该项目持续进行
- 最终会收成“结果汇总”页面和 PDF，不需要评委自己从一堆卡片里拼答案

## 评委常见问题的回答口径

### 1. 你们为什么不是直接堆很多 skills？

因为 skills 适合做能力模块，但科研项目需要：

- 项目级状态
- 项目级知识库
- 结构化成果
- 持久化记录

所以引路虾是产品系统，OpenClaw 是底层执行设施。

### 2. OpenClaw 在这里到底做了什么？

它负责角色执行和 workspace skills 运行。工作台发出动作后，FastAPI 会按需调用 OpenClaw，本地执行角色，再把结果回写成持久化 Markdown。

### 3. 这是不是只是在总结论文？

不是。它最终要解决的是：

- 这个方向现在做到哪一步
- 还有什么值得研究
- 下一步怎么做
- 该如何把一轮讨论沉淀下来

### 4. 数据和安全怎么处理？

当前是本地运行，不暴露公网 OpenClaw。项目数据存本地 SQLite，文献和生成内容围绕项目隔离保存。

### 5. 右侧为什么现在默认可以收起？

因为右侧内容是第二层信息：

- 页面说明
- 当前课题辅助信息
- 研究摘要
- 最近生成内容

它对复盘很有用，但不应该压住中间主舞台。所以现在默认更强调中间工作区，需要时再展开右侧辅助区。

## 当前版本的真实边界

- OpenClaw 当前是按需 CLI 执行，不是常驻通道服务
- PDF 已可入库与检索，但还不是最终的 GROBID / PaperQA 级证据链
- Bohrium skills 已接入工作区，但实际检索仍需可用额度；当前系统允许回退到其他来源
